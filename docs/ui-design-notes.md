# UI 設計メモ(v1.2、2026-07-02)

Phase 2b (Cinny fork) で目指す画面の合意事項。スパイク項目1の気づきから発展し、
モックアップ2種(レイアウト案・ドッキングシェル)ですり合わせて確定。

## 基本方針

- **配信特化・カメラ非対応**: 提供するのは音声 + 画面共有(配信)のみ。
  実装はカメラ UI の非表示化にとどめる(機能削除まではしない — upstream 差分を小さく保つ)
- **配信の視聴はオプトイン**: 強制参加(自動再生)ではなく、見るかどうかを視聴者が選ぶ。
  見ていない配信は購読しない(帯域面でも合理的 — simulcast 購読自体を止める)

## 通話画面(element-call fork 側)

- **表示モードは3段構え**(切替は通話画面上のボタンで行う):
  1. **注視(基本・既定)**: Discord 準拠。1配信を大きく表示 + 他はミニタイル。
     追加 UI は「ミニタイルを隠す」ボタンのみ(メイン配信のみの表示に切替可)
  2. **グリッド(追加オプション)**: 複数配信を同時に見たい場合の均等表示
  3. **複数注視(グリッドの発展)**: グリッド中に「複数注視」ボタンで2枚を選択 →
     2画面並列表示 + 残りはミニタイル(モック C案の挙動)
- グリッド中に通常の「注視」で1枚選べば注視モードへ戻る
- **ミニタイルは実映像を流し続ける**(LiveKit simulcast 低解像度層を購読)
- **ミニタイルの位置はユーザー選択可**: 上・下・左・右
- **話者表示は Discord StreamKit Overlay 風**: アバター+名前ピルを配信画面にオーバーレイ。
  発話中は緑リング+名前ハイライト、ミュートはマイクオフアイコン。
  **オーバーレイはドラッグで移動可能**(これのみ直接操作、他の配置設定は設定画面)
- **配信ストリームのポップアウト(Discord 方式)対応**: タイル単位で個別配信を別ウィンドウの
  ビューア窓に出す。通話はメインウィンドウに残るため**再接続なし**(2026-07-02 実証済み、
  popout-spike.md 追加検証参照)。実装は EC fork のタイル UI に「ポップアウト」ボタン +
  `track.attach()` でビューア窓へ接続。主目的「配信を見ながら他サーバー確認」はこれで満たす
- 通話全体のポップアウト(再接続 ~1.4 秒、cinny 側スパイク実装済み)はオプション扱い。
  マルチモニタで通話 UI ごと移す用途が出たら採用を再検討

## シェル(cinny fork 側)

- **サーバー一覧枠・チャンネル一覧枠とも 上・下・左・右 から選択可**
  (上下配置時は横並び UI。チャンネル一覧はチップ列)
- **配置の設定 UI は設定画面に置く**(ドラッグハンドル方式にはしない)

## デフォルト(初見の友達が見る形)

- **初回参加時(ルーム参加時)に配置セットアップを促す**
- スキップ/未選択時のフォールバックは **Discord 風**: サーバー枠・チャンネル枠は左、
  ミニタイルは下、表示モードは注視(Discord と同じ挙動線)

## 実装マッピング

| 対象 | fork | 対象コード(スパイクで特定済みの周辺) |
|---|---|---|
| 表示モード・注視選択・ミニタイル・隠すボタン・話者オーバーレイ・カメラUI非表示 | element-call | SpotlightTile / grid 系 UI(src/tile/, src/grid/) |
| 配信オプトイン視聴(購読制御) | element-call | LiveKit track subscription 制御 |
| メディアパラメータ(高画質化) | element-call | src/livekit/options.ts(スパイク項目2で実証済み) |
| サーバー枠・チャンネル枠のドッキング | cinny | ClientLayout 系。DOM 構造は upstream のまま CSS(flex-direction / order)+設定切替で実現 |
| 通話画面ポップアウト | cinny | CallEmbed(iframe widget)を別ウィンドウへ再ホスト。要技術検証 |
| 配置設定の保存 | cinny | Matrix account data(端末間同期が効く) |

## リスク・注意

- ドッキング構想は fork-strategy.md「差分は小さく」と緊張関係 →
  レイアウト層の追加 + CSS 切替で実装し、upstream 追従の衝突面を最小化する
- EC のタイル UI は upstream でも活発に変わる箇所 → 新レイアウトは別コンポーネント追加で
  既存を温存する
- **ポップアウトは技術検証が必要**(widget の別ウィンドウ再ホスト、E2EE セッションの扱い)。
  Phase 2b の早い段階で小さく検証すること
- 初回セットアップの促しは「うるさくない」形にする(モーダル連発にしない)
